# 性能分析方法
## 分析卡顿源头
一般卡服都分以下类型:
- 资源不足：资源不足，服务器没内存处理请求
- 实体过多：实体过多，每帧计算量过大
- PT过多：PT多的给服务器内存吃完了
- 人数过多：人数过多，计算量过大
- 有傻逼在卡服：我去你妈的闲jb的没事干卡服干嘛
- 有傻逼在攻击：妈死了的人在发大量请求包，把服务器宽带搞没

## 性能占用分析
当服务器TPS（每秒刻数）出现下降时，及时进行性能占用分析能够快速定位卡顿源头

**`/tps`** - 后台执行该指令后，系统会返回服务器的时时TPS和平均TPS  
**`/mspt`** - 后台执行该指令后，系统会返回服务器的实时MSPT和平均MSPT  
**`/testfor`** - 执行该指令后，系统会返回玩家附近128格的实体数量  
  
**`/timing [分析时长: 数字]`**  
执行该命令后，系统会返回详细的性能占用数据报告
  
| 占用项名称 | 功能说明 |
| :--------- | :------- |
| `ActorLegacyTickSystem` | 处理实体的基础逻辑（如生物、掉落物等） |
| `LevelChunkTickingSystem` | 处理区块的加载和更新（如地形、红石电路等） |
| `GoalSelectorSystem` | 控制生物的人工智能行为（如僵尸追踪、村民路径寻找等） |
| `SendActorDirtyData` | 负责将实体数据同步到客户端 |
| `ActorMotionSyncSystem` | 负责同步实体的移动状态（如生物行走、飞行等） |
  
> 提示：如果对分析结果有疑问，可以尝试使用AI工具辅助解读
  
## 服务健康状态检查
通过性能分析命令可以全面了解服务器的运行状况，及时发现潜在问题  
**执行此命令后玩家也会看到性能分析，尽量低频率执行以免吵到别人**  
  
**`/prof [模式: normal|entity|chunk|pt] [分析时长: 数字]`**
  
| 分析模式 | 功能描述 |
| :------- | :------- |
| `normal` | 标准模式，下面三个模式都占一点，适合日常维护排查 |
| `entity` | 实体分析模式，列出所有已加载实体及其在每个服务器刻（Tick）中的处理时间占用 |
| `chunk` | 区块分析模式，显示所有已加载区块及其在每个服务器刻中的处理时间分布 |
| `pt` | 计划刻分析模式，识别所有有计划刻（Pending Tick）的区块，用于处理如红石电路、作物生长等延迟发生的方块更新事件 |

# 性能优化方法
首先，[分析卡顿源头](性能相关方法.md#分析卡顿源头)，找出哪里导致的卡顿  
现在，根据卡顿源头，进行性能优化

## 资源不足
重启

## 实体过多
执行`/cleaner clean`启动实体清理
执行`/sinfo`看看有多少实体
- 小于 500：应该不是实体问题，找其他源头
- 大于 500：谁在挖矿？
- 大于 1000：有人卡服或刷沙机没关

## PT过多
执行`/prof pt`，系统会返回一些区块坐  
输入最多PT的区块坐标到下面的命令中（不要想着直接传送）  
执行`/om tpch <区块坐标X> <区块坐标Z>`  
现在已经到了导致卡顿的区块了，环顾四周看看什么东西在卡  
- 纯水/岩浆：ojang屎山代码发力了，方块更新一下即可解决
- 大型红石机器：判断是不是卡服鸡，是就拆掉，不是就提醒
- 一切正常，但还在卡：
  - 输入`/log pt`看具体坐标，这个可以直接TP

区块都排查完了 还在卡？  
- 输入`/func maxpt 10000` 释放PT 10秒
- 释放完成后输入`/func maxpt 5` 切换回正常PT
- 还在卡？直接输入`/func maxpt 1` 限制到最低

## 人数过多
无法根治，输入`/func maxpt 1` 限制最大PT数，人少时再放回来

## 有傻逼在卡服
首先分析，他是无意卡服还是恶意卡服
- 无意卡服
  - 进服务器找到这个玩家，提示一下他，提示4次后还不改按恶意卡服处理
- 恶意卡服
  - 拉黑账号和IP，拆除卡服装置，然后给他妈撒了
  - 清除他的玩家NBT数据，释放内存

## 有傻逼在攻击
打开抓包工具，分析服务器发包情况，找出攻击源头，拉黑IP，然后给他妈撒了
